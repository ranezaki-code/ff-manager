## 概要

スプレッドシートの複雑な関数を完全に置き換える、Webアプリ形式のFF作成管理システム。

## 技術仕様

- 形式：シングルHTMLファイル（オフライン動作可能）
- データ保存：localStorageに自動保存
- 対応端末：PC（1280px基準）・スマホ（375px〜）
- レスポンシブ：横スクロールは絶対に発生させない
  - viewport meta タグ必須：<meta name="viewport" content="width=device-width, initial-scale=1.0">
  - 全要素にbox-sizing: border-boxを適用
  - テーブルはtable-layout: fixedで固定幅
  - overflow-x: hiddenを全コンテナに設定

## 画面①：実績入力

### 入力項目（商品ごとに1行）

| 項目 | 内容 |
|------|------|
| 商品名 | テキスト（②商品設定から自動生成） |
| カテゴリ | ホットFF / ホットFF新商品 / 常温FF / 常温FF新商品 / 中華まん / 中華まん新商品 |
| 定価販売数 | 数値入力（週合計） |
| 値引販売数 | 数値入力（週合計） |
| 廃棄数 | 数値入力（週合計） |

### リアルタイム自動計算表示（入力と同時に更新）

入力値から以下を自動計算して商品ごとに表示：

```
販売高 = 定価販売数 × 店頭売価 ＋ 値引販売数 × (店頭売価 − 値引額)
廃棄高 = 廃棄数 × 原価
値引高 = 値引販売数 × 値引額
売変率 = 値引高 ÷ 販売高 × 100（%表示）
利益高 = 販売高 − 廃棄高 − 値引高 − (定価販売数 + 値引販売数) × 原価
利益率 = 利益高 ÷ 販売高 × 100（%表示）
```

```
販売率 = (定価販売数 + 値引販売数) ÷ 作成数 × 100
作成数 = 定価販売数 + 値引販売数 + 廃棄数
目標：96%以上（96%未満は警告表示）
```

表示項目（商品ごとに全て表示すること）：
購入率・販売高・廃棄高・値引高・売変率・利益高・利益率・**販売率**
（利益高・販売率は必ず表示すること。省略禁止）

- 利益率が50%を下回る商品は赤色で警告表示
- カテゴリごとの合計行も表示（販売高・廃棄高・値引高・利益高・利益率）
- 週合計サマリーを表の上部に大きく表示
  （販売高合計・廃棄高合計・値引高合計・利益高合計・利益率・**販売率**）

### 曜日別・時間帯別客数入力

直近4週平均の曜日別・時間帯別客数を入力するグリッド表。

#### 入力形式

**行=曜日（火〜月）、列=時間帯（0時〜23時）** のグリッド表：

| 曜日 | 0時 | 1時 | 2時 | ... | 7時 | 8時 | ... | 22時 | 23時 | 日計 |
|------|-----|-----|-----|-----|-----|-----|-----|------|------|------|
| 火   | 2   | 1   | 1   | ... | 28  | 31  | ... | 15   | 3    | 380  |
| 水   | 2   | 1   | 1   | ... | 28  | 32  | ... | 16   | 4    | 390  |
| ...  |     |     |     |     |     |     |     |      |      |      |
| 合計 | 15  | 9   | 7   | ... | 207 | 235 | ... | 78   | 32   | 2906 |

- 行：曜日（火〜月の7行）
- 列：時間帯（0:00〜23:00 の24列）
- 各セルは直接数値入力可能
- 右端に曜日別日計を自動表示
- 最下行に時間帯別合計を自動表示
- 右下に週総客数を自動表示
- 入力値はlocalStorageに自動保存
- テーブルは横スクロール対応（min-width: 1400px）
- 曜日列は左端にスティッキー固定
- 「全て0にリセット」ボタンで全セルを0にリセット（確認ダイアログあり）

#### キーボード入力の操作性

| キー | 動作 |
|------|------|
| Tab | 右のセルへ移動 |
| Enter | 下の曜日（同じ時間帯）へ移動 |
| ↑ | 上の曜日へ移動 |
| ↓ | 下の曜日へ移動 |
| ← | 左の時間帯へ移動（行末は前の曜日の末尾へ） |
| → | 右の時間帯へ移動（行末は次の曜日の先頭へ） |

- フォーカス時に既存値を自動全選択（すぐ上書き入力可能）
- 入力のたびに日計・時間帯合計・週総客数をリアルタイム更新（グリッド全体の再描画なし・フォーカス維持）
- 画面上部に操作ヒントを表示

## 週別データ管理

実績入力・客数グリッド・チェックリストのデータを週ごとに保存・切り替えする機能。

### 週セレクタ（画面①実績入力タブ上部に常時表示）

```
📅 週  [4/21(火)週] ×   [4/28(火)週 ✓] ×   [＋ 追加]
```

- 登録済みの週が横並びのボタンで表示される
- アクティブ週には「✓」マーク・強調スタイル
- 各週の右に削除ボタン「×」（最後の1週は削除不可）
- 「＋ 追加」ボタンで新しい週を追加

### 週の追加

- 「＋ 追加」タップでモーダル表示
- デフォルト日付：最後の週の次の火曜日を自動計算
- 火曜日以外の日付を入力した場合は確認ダイアログを表示
- 週キーは `YYYY.M.D` 形式（例: `2026.4.28`）
- 週データ構造：`{ ji:{}, cust:[][], checks:{}, weekStart:'YYYY-MM-DD' }`

### 週の切り替え

- ボタンタップでアクティブ週を切り替え
- 切り替え時に現在のデータを保存してから新しい週を読み込む

### 週の削除（2タップ方式）

- 1タップ目：「×」→「削除?」（赤背景・点滅）に変化
- 2タップ目：実際に削除を実行
- 3秒間操作しない場合：自動でキャンセル（「×」に戻る）
- 最後の1週は削除不可（alert表示）

### 週の始まり

- **必ず火曜日** を週の始まりとする（業務スケジュール上）
- 起動時に既存データの週開始日が火曜日でない場合、自動で直近火曜日に移行する（マイグレーション）
- `getRecentTuesday(dateStr)`: 指定日以前で最も近い火曜日を返す
- `getNextTuesday(dateStr)`: 指定日以降で次の火曜日を返す（今日が火曜の場合は7日後）

### データ保存

- 週データは `S.weeks` オブジェクトに週キーをキーとして保存
- 画面切り替え・入力時に自動で `syncActiveWeek()` が呼ばれアクティブ週に書き込み
- localStorage キー `ffv6` に全週データを一括保存

## 画面②：商品設定

### カテゴリON/OFFトグル

カテゴリごとにON/OFFトグルを設置：
- ホットFF
- ホットFF新商品
- 常温FF
- 常温FF新商品
- 中華まん（冬季のみON・それ以外はOFF）
- 中華まん新商品（冬季のみON・それ以外はOFF）

OFFにした場合の動作：
1. ④作成表からそのカテゴリを完全に非表示にする
2. OFFカテゴリの予算を残りのカテゴリに比率で自動再配分
3. 再配分後の予算で作成数を自動再計算
4. トグル切り替えと同時にリアルタイムで全画面更新

### 商品削除時のクリーンアップ（重要）

商品を削除する際は、以下の孤立データを同時に削除する：

| 削除対象 | キー |
|---------|------|
| アクティブ週の実績 | `S.ji[pid]` |
| 全週の実績 | `S.weeks[*].ji[pid]` |
| チェック状態 | `S.checks` のキーが `*_pid` で終わるもの |
| 新商品予算 | `S.budget.newProductBudgets[pid]` |
| 値引き履歴 | 削除しない（名前付きで保存済みのため履歴は保持） |

```javascript
function delProduct(pid){
  if(!confirm('この商品を削除しますか？'))return;
  S.PM = S.PM.filter(p => p.id !== pid);
  // 孤立データをクリーンアップ
  delete S.ji[pid];
  Object.values(S.weeks||{}).forEach(w => { if(w.ji) delete w.ji[pid]; });
  Object.keys(S.checks).forEach(k => { if(k.endsWith('_'+pid)) delete S.checks[k]; });
  if(S.budget.newProductBudgets) delete S.budget.newProductBudgets[pid];
  saveS(); renderProductCards();
}
```

### カテゴリ別の作成時間・販売許容時間設定

カテゴリごとに以下を設定する：

| カテゴリ | 販売許容時間 | 作成回数 | デフォルト作成時間 |
|---------|-----------|---------|----------------|
| ホットFF | 6時間 | 1日4回 | 6:00/10:00/14:00/17:00 |
| ホットFF新商品 | 6時間 | 1日4回 | 6:00/10:00/14:00/17:00 |
| 常温FF | 10時間 | 1日1回 | 10:00のみ |
| 常温FF新商品 | 10時間 | 1日1回 | 10:00のみ |
| 中華まん | 10時間 | 1日2回 | 6:00/10:00のみ |
| 中華まん新商品 | 10時間 | 1日2回 | 6:00/10:00のみ |

設定項目：
- 販売許容時間（時間）：数値入力で自由に変更可能
- 作成時間：ON/OFFトグルで選択（6:00/10:00/14:00/17:00）
  └ 上記4つ以外の時間を使いたい場合は「＋カスタム時間を追加」で任意の時間を追加可能
  └ 例：7:00に作成したい場合 → 「7:00」と入力して追加
  └ 追加した時間はトグルで個別にON/OFF可能
  └ 不要になった時間は削除ボタンで削除可能
- 値引き開始 = 廃棄時間 - 2時間（自動計算・表示のみ）

ホットFFの設計思想：
  販売許容6時間 → 4時間分作成 + 残2時間で値引き完売
  → 1日4回の作成が成立する

常温FF・中華まんの設計思想：
  販売許容10時間 → 1〜2回のまとめ作成が可能
  → 作成回数を減らして業務負担を軽減

### 当週の施策

本部からの販促・新商品発売・キャンペーンなどを登録するセクション。
登録した施策は補正係数の自動提案ロジックに反映される。

#### 施策の入力項目

| 項目 | 内容 |
|------|------|
| 施策名 | テキスト入力（例：からあげクン新フレーバー発売） |
| 適用期間 | 全日 / 平日（火〜金・月）/ 週末（土・日）/ 個別曜日（火〜月） |
| 想定インパクト | 数値入力（%）。プラスで増産・マイナスで減産。色分け表示 |

#### データ構造

```javascript
S.施策 = [
  {
    id: 'sch_1745000000000',  // Date.now()で生成
    name: 'からあげクン新フレーバー発売',
    days: 'all',   // 'all' | 'weekday' | 'weekend' | '火'|'水'|'木'|'金'|'土'|'日'|'月'
    impact: 15     // % (正の値=増産、負の値=減産)
  }
]
```

適用曜日の対応表：

| days 値 | 適用曜日 | DAYSインデックス |
|---------|---------|----------------|
| `'all'` | 全日 | 0〜6 |
| `'weekday'` | 平日（火〜金・月） | 0,1,2,3,6 |
| `'weekend'` | 週末（土・日） | 4,5 |
| `'火'`〜`'月'` | 個別指定 | 対応インデックス |

### 補正係数の自動提案（2種類）

#### ① 単日提案ボタン（補正係数フィールド横）

各曜日の「補正係数(%)」フィールドの右に「🤖 提案」ボタンを設置。
タップ1回で当日の施策・イベント・天気を加味した値を自動入力する。

#### ② 全曜日一括提案ボタン

当週の施策セクションの下部に「🤖 全曜日の補正係数を一括提案」ボタンを設置。
クリックするとモーダルで7曜日分の提案値を一覧表示し、手動調整後に一括適用できる。

```
┌──────────────────────────────────────────┐
│ 🤖 補正係数の提案                          │
│ 📐 ルールベース（AI: クレジット不足）        │  ← AIまたはルールベースを明示
│                                            │
│ 火  [ 20 ] %   +8% → +20%                 │
│ 水  [ 15 ] %   0% → +15%                  │
│ 木  [ 15 ] %   0% → +15%                  │
│ 金  [ 15 ] %   0% → +15%                  │
│ 土  [ 25 ] %   0% → +25%                  │
│ 日  [ 25 ] %   0% → +25%                  │
│ 月  [ 15 ] %   0% → +15%                  │
│                                            │
│ [キャンセル]           [✓ 一括適用]         │
└──────────────────────────────────────────┘
```

- 各行の数値は手動調整可能
- 現在値との差分を「現在0% → +15%」形式で右列に表示
- 「✓ 一括適用」で全7曜日の補正係数を一括更新

#### 提案ロジック

**優先①：Claude API（APIキー設定済みの場合）**
```
プロンプトに以下を含めてAIに判断させる：
- 当週の施策リスト（名前・適用期間・想定インパクト）
- 各曜日の状況（天気・気温・イベント）
- 指示：JSON形式 {"火":数値,"水":数値,...,"理由":"簡潔に"} で返答
```

**優先②：ルールベース（APIキー未設定 / APIエラー時のフォールバック）**
```
補正係数 = 以下の合算（曜日ごとに計算）

1. 施策インパクト合算
   → 当日に適用される施策の impact をすべて合算

2. 天気補正
   → 雨: -5%  /  雪: -10%  /  台風: -25%

3. イベント補正
   → イベント欄が空でない場合: +5%

4. 気温補正
   → 5℃以下: -5%  /  33℃以上: -5%
```

計算例：
```
火曜（施策: からあげクン+15%、イベント: 上高地あり、天気: 晴れ）
→ +15% + 5% = +20%

土曜（施策: からあげクン+15% + 週末キャンペーン+10%）
→ +15% + 10% = +25%
```

## 画面④：作成表（現場使用メイン画面）

### ⚠️ 重要：補正係数の適用ルール

**補正係数は「週の目標販売数（週合計）」に対して適用する。**
時間帯ごとの作成数に直接掛け算してはいけない。

正しい計算フロー：
```
① 週の目標販売数（ベース）を算出
② 週の目標販売数 × (1 + 補正係数) = 補正後週目標
③ 補正後週目標 × 時間比率 = 各時間帯の作成数
④ 小数丸め処理（0.3以上→1個、0.3未満→0個）
```

間違った計算（禁止）：
```
× 各時間帯の作成数 × (1 + 補正係数)  ← これは金額が狂う
```

### セクション2：来週の予算計画（④作成表から自動算出）

表示する10項目：

| 項目 | 説明 |
|------|------|
| 次週作成高（正式） | ④作成表の計画値から自動取得 |
| 想定廃棄額（売価ベース） | 商品別廃棄率 × 作成数 × 売価（商品ごとに計算後集計） |
| 想定売変額 | 商品別値引率 × 作成数 × p.disc（商品ごとのp.discを使用） |
| 想定売変率 | 想定売変額 ÷ 次週売上高（正式）× 100 |
| 次週売上高（正式） | 次週作成高（正式）− 想定廃棄額 − 想定売変額 |
| 新商品予算（最大値） | max(0, 次週売上高 − 原価合計)（赤字にならない投資上限） |
| 新商品予算（正式） | 商品ごとの手入力合計（S.budget.newProductBudgets） |
| 次週利益高 | 次週売上高（正式）− 原価合計 − 新商品予算（正式） |
| 次週利益率 | 次週利益高 ÷ 次週売上高（正式）× 100 |
| 実績 適正販売率（全週累計） | 全週合算の定価販売数 ÷ 全週合算の作成数（廃棄率算出の根拠として表示） |

**各項目の計算式（全自動・商品ごとの実績レートを使用）：**
```
// ① 作成計画から直接集計
次週作成高（正式）   = Σ getProdCnt(p, 曜日, スロット) × 店頭売価
次週原価合計         = Σ getProdCnt(p, 曜日, スロット) × 原価

// ② 商品別の実績廃棄率・値引率（全週通算）
ji_p = _getJiAllWeeks(pid)
     = 当週 S.ji[pid] + 他週 S.weeks[*].ji[pid] を合算
廃棄率_p = ji_p.total > 0 ? ji_p.haiki / ji_p.total : 1.0  ← 実績なし=100%（保守的）
値引率_p = ji_p.total > 0 ? ji_p.nebiki / ji_p.total : 0    ← 実績なし=0%

// 【重要】ji_p.total = 0（実績未入力）の場合は廃棄率100%で計算する。
//
// 作成計画がある以上、実績のない商品は「全廃棄リスク」として見積もるのが正当。
// 利益がマイナスになるのは意図した動作（警告として機能する）。
//
//   → 「このカテゴリを作り続けると赤字になる」というシグナル
//   → 対処法：①実績入力で実績を入れる  または  カテゴリをOFFにする
//
// 実績なし商品がある場合は予算計画に警告バナーを表示する：
// 「⚠️ 実績データなし商品X件：廃棄率100%で計算しています」

// ③ 次週想定値（商品ごとに計算して集計）
// 廃棄は売価ベース（作成高からの引き算で売上高が正確に求まる）
想定廃棄額   = Σ getProdCnt × 廃棄率_p × 店頭売価
// 売変額は商品別値引単価（p.disc）を使用
想定売変額   = Σ getProdCnt × 値引率_p × p.disc

次週売上高（正式）  = 次週作成高 − 想定廃棄額 − 想定売変額
想定売変率         = 想定売変額 ÷ 次週売上高（正式）× 100
新商品予算（最大値）= max(0, 次週売上高 − 次週原価合計)

// ④ 利益（新商品予算を差し引く）
// 新商品予算（正式）は全廃棄になっても利益マイナスにならないよう確保
次週利益高  = 次週売上高（正式）− 次週原価合計 − 新商品予算（正式）
次週利益率  = 次週利益高 ÷ 次週売上高（正式）× 100

新商品予算（正式）= Σ S.budget.newProductBudgets[商品ID]（商品別手入力）
```

**廃棄額が「売価ベース」である理由：**
```
次週売上高 = 次週作成高 − 想定廃棄額 − 想定売変額
= cnt × 売価 − cnt × 廃棄率 × 売価 − cnt × 値引率 × 値引単価
= cnt × 売価 × (1 − 廃棄率) − cnt × 値引率 × 値引単価
= (定価販売見込) × 売価 + (値引販売見込) × (売価 − 値引単価)
= 実際の予想売上高  ← 数学的に証明済み
```
※実績タブの廃棄高（`haiki × 原価`・原価ベース）とは基準が異なる。
　予算計画は売上高計算のために売価ベースを使用し、ラベルに「売価ベース」と明記する。

**なぜ商品ごとの実績レートを使うのか：**
- 作成計画はすでに「適正販売率」を考慮した数量になっている
- その作成計画に対して商品ごとの実績廃棄率・値引率を当てることで
  より精度の高い次週予算を算出できる

**新商品予算（正式）— 商品別手入力：**
- 「新商品」で終わるカテゴリの商品のみ一覧表示
  （例：ホットFF新商品、常温FF新商品、中華まん新商品）
- 商品ごとに予算額を手入力（数値入力・100円刻み）
- 合計が新商品予算（最大値）を超えると赤字で警告（⚠ 最大値超過）
- データは `S.budget.newProductBudgets = { 商品ID: 金額 }` に保存

**週次予算設定（手動）：**
| 項目 | 説明 |
|------|------|
| 作成高予算（円） | 目標作成高を手入力 |
| 廃棄予算（円） | 廃棄の上限予算を手入力 |

## 計算ロジック（完全版）

### 1. 購入率
```
購入率 = 定価販売数 ÷ 週総客数（直近4週平均）
```

### 2. 標準偏差（客数の波）
```
1. 週総客数 ÷ 7 = 1日平均客数
2. 各曜日の客数 − 1日平均客数 = 差
3. 差を2乗 → 全曜日合計 ÷ 7 → √ = 標準偏差
```

### 3. 強気度係数
```
値入額 = 店頭売価 − 原価
強気度係数 = 値入額 ÷（原価 + 値入額）
```

### 3-2. 値引額の決め方（⑥値引額設定タブで使用）
```
① 原価 × 0.25 = 損益分岐流出率相当額
② その金額が利益額（販売高 − 原価合計）を超えていないか確認
③ 超えていない範囲で5円単位に丸めた金額を値引額にする

例：からあげクンレギュラー
  原価：115.65円
  115.65 × 0.25 = 28.9円 → 5円単位で→ 29円（値引額）
```

### 4. ゆとり個数
```
ゆとり = 標準偏差 × 購入率 × 強気度係数
```

### 5. 週の目標販売数（ベース）
```
週の目標 = 定価販売数 + ゆとり
```

### 6. 最低作成数の適用
```
if 週の目標 < 最低作成数 × 7:
    週の目標 = 最低作成数 × 7
```

### 7. 外部ファクト補正（週合計に対して適用）
```
補正後週目標 = 週の目標 × (1 + 補正係数)
※補正係数は曜日ごとに設定。その曜日の週目標分配後に適用。
```

### 8. 曜日別・時間帯別作成数

⚠️ 重要：時間比率の計算は「週総客数（全時間帯）」を分母にする

```
時間比率 = その時間帯の客数 ÷ 週総客数（全時間帯合計）

算出範囲内の時間帯の場合：
  作成数（小数） = 補正後週目標 × 時間比率

算出範囲外の時間帯の場合：
  作成数 = 0（最低作成数が設定されていれば最低作成数を適用）

小数丸めルール：
  0.3以上 → 1個（切り上げ）
  0.3未満 → 0個（切り捨て）
```

### 8-2. 客数割合の3区分（作成時間帯の判断に使用）

PDFの客数割合1・2・3に対応：

```
客数割合1（4ブロック別）：
  ブロックA = 6:00〜9:59の客数合計 ÷ 算出範囲内総客数
  ブロックB = 10:00〜13:59の客数合計 ÷ 算出範囲内総客数
  ブロックC = 14:00〜17:59の客数合計 ÷ 算出範囲内総客数
  ブロックD = 18:00〜の客数合計 ÷ 算出範囲内総客数

客数割合2（全体）：
  算出範囲内の総客数（100%基準）

客数割合3（午前・午後2分割）：
  算出範囲内の客数を午前/午後で分割した割合
```

これらは計算ロジックの内部処理でのみ使用する。
画面には表示しない。
②商品設定の「最適作成時間の提案」機能の計算に活用する。

### 9. 撤退ライン判定
```
原価率 = 原価 ÷ 店頭売価
撤退客数 = 原価率 ÷ 購入率

単一時間帯の場合：
  その時間帯の客数 < 撤退客数 → 作成数を0にする

まとめ作成（複数時間帯合算）の場合：
  合算する時間帯の客数合計 < 撤退客数 → 作成しない
  合算することで撤退ラインを突破できれば → まとめて作成する

※最低作成数が設定されている場合は撤退ライン判定より優先
※深夜帯はまとめ作成で撤退ラインを突破できるか特に注意
```

### 10. 金額計算（①実績入力・⑤予算管理で使用）
```
販売高 = 定価販売数 × 店頭売価 ＋ 値引販売数 × (店頭売価 − 値引額)
廃棄高 = 廃棄数 × 原価
値引高 = 値引販売数 × 値引額
売変率 = 値引高 ÷ 販売高 × 100
利益高 = 販売高 − 廃棄高 − 値引高 − (定価販売数 + 値引販売数) × 原価
利益率 = 利益高 ÷ 販売高 × 100
```

### 11. 値引き目安時刻の算出（在庫入力不要）
```
廃棄時間 = 作成時間 + 販売許容時間
値引き目安時刻 = 廃棄時間 - 2時間

例（ホットFF・販売許容6時間）：
  6:00作成 → 廃棄12:00 → 値引き目安 10:00
  10:00作成 → 廃棄16:00 → 値引き目安 14:00
  14:00作成 → 廃棄20:00 → 値引き目安 18:00
  17:00作成 → 廃棄23:00 → 値引き目安 21:00

例（常温FF・販売許容10時間）：
  10:00作成 → 廃棄20:00 → 値引き目安 18:00

※実際の値引き数はAICO（別タブレット）で在庫確認して判断する
※このシステムでは「いつ値引きするか」の目安時刻のみを提供する
```

## デザイン指針（現場目線・直感的・シンプル）

### 基本方針
- **現場のクルーが初めて見ても迷わず使える**ことを最優先
- 色は意味を持たせる（赤=警告・要注意、緑=完了・OK、青=情報、グレー=無効）
- 1画面に情報を詰め込みすぎない
- 文字は大きめ（本文14px以上、数値は16px以上）

### カラーパレット
- メインカラー：#1E3A5F（濃い紺）→ ヘッダー・重要ボタン
- アクセントカラー：#2196F3（青）→ 作成数・情報
- 成功：#4CAF50（緑）→ チェック完了・OK
- 警告：#FF5722（赤）→ 利益率低下・予算オーバー・要注意
- 新商品：#FF9800（オレンジ）→ 新商品バッジ
- 背景：#F5F5F5（薄いグレー）→ ページ背景
- カード背景：#FFFFFF（白）→ コンテンツエリア

### 作成表のデザイン
- 作成数は大きく太字で表示（視認性最優先）
- チェック済みセル：グレーアウトして一目で「完了」とわかる
- 補正バッジ：小さく目立たないデザイン（メインは作成数）
- 曜日タブ：今日の曜日をハイライト表示
- 特需日（補正+10%以上）は曜日タブを赤系でハイライト

### ボタンのデザイン
- 主要アクション（全完了・保存等）：大きく・メインカラー
- 補助アクション（リセット等）：小さく・グレー系
- 危険なアクション（削除等）：赤系
- タップ領域は最低44×44px（スマホ対応）

### アイコン・視覚要素
- チェックボックス：チェック済みは緑のチェックマーク
- 新商品バッジ：「新」の赤バッジ（目立つが邪魔にならないサイズ）
- 警告：⚠️アイコン＋赤文字
- 完了率：シンプルなプログレスバー

## データ保存仕様

- localStorageに自動保存
- 週ごとにデータを保持（過去8週分）
- エクスポート：JSONでダウンロード可能（全店舗・全週データを一括出力）
- インポート：JSONファイルを読み込んで全データを復元（確認ダイアログあり・読み込み後リロード）

### エクスポート・インポート仕様

#### エクスポート（exportData）
```javascript
// 出力形式
{
  version: 'ffv6',
  exportedAt: ISO文字列,
  stores: storeList,              // 店舗一覧
  currentStoreId: string,         // 現在の店舗ID
  storeData: { [store_id]: {...} } // 各店舗のlocalStorageデータ
}
```
- ファイル名：`FF管理データ_YYYYMMDD.json`
- 全店舗・全週データを一括でダウンロード

#### インポート（importData）
- `.json` ファイルを選択して読み込み
- `version: 'ffv6'` の確認（バージョン不一致はエラー）
- 確認ダイアログ：「現在のデータは上書きされます」
- 復元対象：stores・各店舗データ・currentStoreId
- 読み込み後：`location.reload()` で画面を再起動

**用途：**
- デバイス間のデータ移行（スマホ ↔ PC）
- ドメイン間のデータ移行（GitHub Pages ↔ ローカル）
- バックアップ・リストア

**③ 作成表への施策・イベント・補正の反映フロー確認（監査結果）**

ユーザーからの質問「実際に作成表にイベントや施策・補正などから分析して個数を反映してくれるのか？」に対して、コードを精査した結果を下記に記録する。

| 入力 | 作成表への反映 | 経路 |
|------|--------------|------|
| 補正係数（③外部ファクト） | ✅ **直接反映** | `getCorrTarget` が `S.ext[dayIdx].correction` を使用 |
| カテゴリ別ルール（③外部ファクト） | ✅ **直接反映** | `getCorrTarget` が `S.rules` を参照し `ruleCorr` を累算 |
| 施策（③外部ファクト） | ⚠️ **間接のみ** | 施策→🤖AI提案→補正係数設定→反映。施策のみでは作成数は変わらない |
| イベント・天気（③外部ファクト） | ⚠️ **AI提案インプットのみ** | 補正係数のAIプロンプトに使用。直接計算には使われない |

**`getCorrTarget` の処理フロー**

```
weekTarget = max(teika + slack, minCount × 7)
↓ 補正係数を適用
corrTarget = weekTarget × (1 + extCorr + ruleCorr / 100)
↓ その日の客数分配
getProdCnt = roundCnt(corrTarget × slotCust / weekTotal)
```

**重要な前提条件：時間帯別客数データが必要**

`getProdCnt` はスロット客数（`getSlotCust`）を使って日別作成数を計算するため、
①実績入力タブの「曜日別・時間帯別客数グリッド」にデータが入力されていないと、
スロット客数=0となり補正係数の効果が計算に現れない（結果は常に `minCount`）。

```
客数グリッド未入力 → slotCust=0 → getProdCnt=max(0, minCount) → 補正効果なし
客数グリッド入力済 → slotCust>0 → 補正係数が乗算で効く → 作成数が変化する
```

実際の検証結果：7〜21時に各20人の客数を仮定した場合、
- 補正0% → からあげクン火曜日 26個
- 補正+30% → 34個（×1.3倍）✅
- 補正-30% → 19個（×0.7倍）✅

**④ 予算の「追加作成数」と作成表の関係**

⑤予算管理タブで入力した「追加作成数（個）」は、財務計画上の投資額管理のためのもので、
**④作成表の個数には自動連動しない**。
作成表は客数グリッド・補正係数・ルールから自動計算されるもの。
投資商品を増産したい場合は、③外部ファクトで補正係数を調整するか、
直接④作成表で個数を増やす。
